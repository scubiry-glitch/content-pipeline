# 内容生产流水线 Content Pipeline

> 参赛项目说明文档

**一句话定位**:一套 AI 驱动的产业研究内容工业化生产系统,把"高质量观点"稳定转化为"双周更新的研究报告",核心差异化在于业界少见的 **BlueTeam 蓝军评审机制** 与 **132 位专家的认知数字孪生 (CDT)**。

## 1. 为什么做这个项目 Why

### 1.1 我们看到的痛点

**深度内容创作者普遍面临同一个矛盾**:拥有强研究与观点输出能力,却缺乏工程化的生产工具链,导致:

1. **知识无法沉淀**——242 份历史研报躺在硬盘里,写新内容时无法被引用、对比、复用;
2. **手工操作割裂**——研究、写作、排版、多态转换分散在 10+ 个工具里,每篇文章重复劳动;
3. **产出无法规模化**——手动流程决定了产出频率不稳定,无法给读者一个双周更新的承诺;
4. **质量靠个人把关**——缺少结构化的评审机制,内容质量波动大。

### 1.2 我们要解决的问题

> 让创作者专注于"思考",而不是"操作工具"。

具体目标:建立一条标准化、可复用的流水线,在保证质量的前提下实现**双周稳定产出**,把历史资产与新研究自动关联成可检索的知识网络。

### 1.3 成功的衡量标准

| 维度 | 指标 | 现状 | 目标 |
|------|------|------|------|
| 质量 | 蓝军评审通过率 | 人工把控 | 100% |
| 频率 | 内容产出周期 | 不定 | 双周更 |
| 效率 | 单期生产时间 | 数天 | < 2 天 |
| 自动化 | 无需人工干预环节 | 0% | > 70% |
| 复用 | 历史数据引用率 | 低 | 每期自动关联历史 |

> 源自 `WHY.md` 项目立项章节。

## 2. 主要客户 Who

### 2.1 核心用户画像:深度内容创作者

| 维度 | 描述 |
|------|------|
| 身份 | 独立分析师、行业研究员、券商/投行撰稿人、产业咨询顾问 |
| 能力 | 深度研究能力 ★★★★★ · 观点输出能力 ★★★★★ · 工程化能力 ★★ |
| 痛点 | 手工切换工具、知识无沉淀、无法稳定产出 |
| 期望 | 把研究精力花在"想清楚",把操作执行交给流水线 |

### 2.2 间接受益者:内容消费者

金融机构从业者、企业管理者、政策研究者、个人投资者——他们为"稳定、可追溯、逻辑严谨"的深度分析付费。

### 2.3 三个典型使用场景

**场景 A — 独立分析师**
> 早上打开系统输入选题"保租房 REITs 市场分析",Planner Agent 生成三层大纲;Research Agent 并行检索 242 份历史研报 + Web 数据,半小时交出研究包;Writer Agent 吐出初稿;BlueTeam 三位虚拟专家并行评审两轮;晚上确认发布,全流程不到 2 天。

**场景 B — 金融机构研究员**
> 通过专家库挑选"房地产金融"领域 3 位虚拟专家,让他们像真实同事一样参与选题评审与写作提示,复用机构内 200+ 份历史报告作为底稿,保证每篇输出符合合规要求。

**场景 C — 读者订阅者**
> 订阅创作者的双周研究,每期内容都自带"关联历史观点演化""事实核查清单""蓝军评审意见",可一键回溯到 18 个月前的同类分析。

## 3. 功能设计 What

### 3.1 核心流水线:四阶段

```
    ┌───────┐   ┌────────┐   ┌────────┐   ┌────────┐
    │ 选题  │──▶│  研究  │──▶│  写作  │──▶│  发布  │
    └───────┘   └────────┘   └────────┘   └────────┘
     RSS聚合     RAG检索      大纲生成      多态转换
     热度评分    Web采集      初稿写作      多平台发布
     事实核查    事实核查      BlueTeam      数据监控
     差异化     信息增量      评审迭代      A/B 测试
```

| 阶段 | 关键能力 |
|------|----------|
| **选题** | RSS 多源聚合 · 热度评分 · 事实核查预检 · 差异化角度建议 · 历史研报复用推荐 |
| **研究** | 历史研报 RAG 检索(pgvector) · Web 深度搜索 · 跨源去重 · 信息增量周期监控 |
| **写作** | 三层结构大纲(议题→论点→论据) · 流式初稿生成 · **BlueTeam 3×3×2 评审** · 多轮迭代优化 |
| **发布** | 多态生成(长文/短文/图文) · 多平台一键发布 · 阅读/互动/转化监控 · 标题封面 A/B |

### 3.2 系统分层架构

```
┌──────────────────────────────────────────────────┐
│  客户端层   Web Dashboard · CLI · API · 运营后台 │
├──────────────────────────────────────────────────┤
│  网关层     Nginx(反向代理 · SSL · 负载均衡)       │
├──────────────────────────────────────────────────┤
│  API 层     Fastify + TypeScript(路由/中间件/插件)│
├──────────────────────────────────────────────────┤
│  业务层     Pipeline · Asset · Topic · Research ··│
│             Expert · ContentLibrary · Publisher   │
├──────────────────────────────────────────────────┤
│  Agent 层   Planner · Researcher · Writer ·       │
│             BlueTeam(蓝军评审)                    │
├──────────────────────────────────────────────────┤
│  引擎层     LLM Router · Embedding · Vector       │
│             Search · Web Search · Document Parser │
├──────────────────────────────────────────────────┤
│  数据层     PostgreSQL + pgvector · Redis ·       │
│             MinIO · BullMQ · FileSystem           │
└──────────────────────────────────────────────────┘
```

### 3.3 四大核心 Agent

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **PlannerAgent** | 选题规划 · 大纲生成 | 选题主题 | 三层大纲 + 数据需求 |
| **ResearchAgent** | 素材检索 · 数据分析 | 大纲 + 关键词 | 研究数据包 + 洞察 |
| **WriterAgent** | 初稿生成 · 引用标注 | 大纲 + 研究数据 | 带引用的初稿 |
| **BlueTeamAgent** | 3×3×2 深度评审 | 初稿 | 评审意见 + 修订稿 |

> 源代码:`api/src/agents/{planner,researcher,writer,blueTeam}.ts`

### 3.4 高级模块

**Content Library v7.0 — 4 层递进检索**

```
L4  跨内容推理层    观点演化追踪 / 因果链 / 矛盾检测
L3  混合检索层      Vector + BM25 + RRF + LLM 重排
L2  层级加载层      L0 摘要 80tok → L1 片段 400tok → L2 全文 1000+tok
L1  知识整合层      事实三元组 / 实体归一化 / 去重合并
```

**Expert Library v5.0 — 132 位专家认知数字孪生**

> 不是简单的 Prompt 模板,而是把专家的 Persona(价值取向)、Method(分析框架)、EMM(决策门控)完整建模,让 AI "像该专家那样思考"。

**Unified Taxonomy v6.3 — 统一分类字典**

> 15 个领域 × 多级映射,打通 RSS 热点 / Assets 素材 / Expert 专家三套分类体系,实现跨数据源联动。

## 4. 创新亮点 How it's Different

### 4.1 BlueTeam 蓝军评审机制 ⭐ 核心差异化

**普通方案**:让一个 LLM 给初稿打分、提建议。
**我们的方案**:**3 位虚拟专家 × 3 个思维维度 × 2 轮迭代**

| 角色 | 关注维度 |
|------|----------|
| **批判者 Challenger** | 逻辑漏洞 · 论证跳跃 · 数据可靠性 · 隐含假设 |
| **拓展者 Expander** | 关联因素 · 国际对比 · 交叉学科 · 长尾效应 |
| **提炼者 Synthesizer** | 核心论点 · 结构优化 · 金句提炼 · 消除冗余 |

**评审流程**:3 专家并行 → 每人 3 个角度 → 每角度 1 个问题 → 2 轮迭代 → 人工最终确认

> 相当于把"独立评审 + 发散补充 + 结构凝聚"三种思维模式内建进流水线,避免单一 LLM 评审的盲区。

### 4.2 Expert Library — 认知数字孪生 (CDT)

**普通方案**:写几条"你是资深房地产分析师"的 Prompt。
**我们的方案**:为每位专家建立完整的 CDT,包含:

- **Persona**:价值取向、思维偏好、标志性表述
- **Method**:分析框架、推理链条、评审流程
- **EMM (Expert Mental Model)**:决策关键因子、权重、一票否决规则
- **盲点自觉**:专家自己不擅长的领域,避免越界发言

132 位专家数据落在 `api/src/modules/expert-library/data/`,可直接在 Web Dashboard 上与任一专家进行 1v1 对话,或作为 BlueTeam 评审人参与流水线。

### 4.3 Content Library — 4 层混合检索

**普通方案**:纯向量搜索,召回一堆相似片段。
**我们的方案**:

| 层级 | 技术 | 收益 |
|------|------|------|
| L3 检索 | Vector + BM25 + RRF + LLM Rerank | 精度较纯向量 ↑ ~40% |
| L2 加载 | 摘要 → 片段 → 全文三级递进 | Token 消耗 ↓ 80–90% |
| L1 整合 | 事实三元组 + 实体归一化 | 从孤立内容到知识网络 |
| L4 推理 | 观点演化 / 因果链 / 矛盾检测 | 跨内容的二阶洞察 |

### 4.4 统一分类字典打通数据孤岛

15 个标准领域 × 多级标签,自动映射到 RSS / Assets / Expert 三个数据源:
- RSS 热点 → 自动匹配专家领域 → 自动分配 BlueTeam 评审人
- Assets 素材 → 自动分类 → 任务推荐时精准关联
- 同一分类内容 → 自动聚合 → 构建知识图谱

### 4.5 多模型 LLM Router 与自动降级

| 任务类型 | 首选模型 | 考量 |
|----------|---------|------|
| 规划 / 分析 / 蓝军评审 | Kimi 长上下文 | 质量优先 |
| 写作 | Kimi / Claude | 中文表达 + 质量 |
| 标签 / Embedding | OpenAI | 成本优先 |

主备链:**Kimi → Claude → OpenAI**,任一环节失败自动降级,保证流水线不断链。

## 7. 快速体验

```bash
# 1. 克隆仓库
git clone <repo-url>
cd content-pipeline

# 2. 配置环境变量(至少填一个 LLM Key)
cp .env.example .env
# 编辑 .env 填入 KIMI_API_KEY

# 3. 一键启动
docker-compose up -d

# 4. 访问
# API:       http://localhost:3000
# Dashboard: http://localhost:8080
```

创建一个研究任务:

```bash
curl -X POST http://localhost:3000/api/v1/production \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-api-key" \
  -d '{"topic":"保租房 REITs 市场分析"}'
```
