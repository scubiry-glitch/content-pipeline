# 多维 Scope 设计方案：项目 · 主题 · 客户

## 1. 现有问题诊断

### 1.1 存量 Scope 概况

| 维度 | kind 值 | 现有 scope 数 | 有绑定的 | 无绑定的 |
|------|---------|--------------|----------|----------|
| 项目 | `project` | 17 | 16 | 1（述职材料） |
| 客户 | `client` | 8 | 7 | 1（城投） |
| 主题 | `topic` | 5 | 5 | 0 |

**绑定分布**：64 场会议有 scope 绑定，仅 1 场未绑定。但——

### 1.2 三个核心问题

**问题 A：topic 维度被产品名占位，丧失了"讨论主题"语义**

现有 topic scope：
- `装修信托` / `装修贷款` / `租金贷款` — 这些是**金融产品名**，不是讨论主题
- `AI` / `AI工具` — 唯一接近"主题"的

结果：42 场会议没有 topic 绑定（64 - 21 = 43 场），因为大多数会议讨论的"风控""供应商""资金结构"没有对应 scope。

**问题 B：client 维度混入了地理分区**

- `上海` / `北京` / `成都` 是贝壳内部的地理事业部，不是外部客户
- 它们作为 `贝壳找房` 的子 scope 存在，但与 `万华` `锦江` `宁波银行` 并列，语义不一致

**问题 C：project 维度过碎，层级不均**

- 17 个 project scope 中，5 个是 `不动产管理` 或 `美租` 的子 scope（长租公寓/酒店物业/行业合作/海外合作、集中式/分散式/设计师）
- 但 `金融合作`(8场)、`业务支持`(10场) 是按功能分组，不是按业务线分组
- 维度混杂：有按产品分的（美租、被窝），有按功能分的（金融合作、业务支持），有按组织分的（集团分析）

### 1.3 影响

- CEO `loadPromptCtx` 按 scope 聚合时，topic 维度几乎空转——风控/供应商/资金结构等高频主题无法被 CEO prompt 感知
- `scopeMatcher` 自动匹配只命中 product 名，无法识别"这场会在讨论风控"
- Panorama 六房间的指标在 topic 维度上失真

---

## 2. 设计原则

| # | 原则 | 说明 |
|---|------|------|
| P1 | **正交性** | 三个维度互相独立：project 回答"做什么业务"，theme 回答"讨论什么问题"，client 回答"和谁合作" |
| P2 | **每维度 5-10 个 scope** | 太少无区分度，太多 CEO prompt ctx 放不下 |
| P3 | **每 scope ≥ 3 场会议** | 低于 3 场的合并到父 scope 或通用 scope |
| P4 | **层级不超过 2 层** | parent_scope_id 最多一层嵌套 |
| P5 | **kind 语义不变** | 继续用 `project`/`client`/`topic`，`topic` 语义从"产品主题"重新定义为"讨论主题" |
| P6 | **one-per-kind 约束保留** | 每场会议每个 kind 最多绑一个 scope，3 个 kind = 3 个维度独立标注 |

## 5. 与 CEO 应用对接

### 5.1 三维 scope 如何被 CEO prompt 消费

当前 `loadPromptCtx` 只接受单个 `scopeId`。三维 scope 下，CEO 需要同时看到三个维度的上下文：

**方案：主 scope + 关联 scope 模式**

```
CEO 选择焦点 scope（通常是一个 project scope）
  → loadPromptCtx 自动 JOIN 同会议的 client/topic scope
  → PromptCtx 扩展字段：

  {
    scopeId: "P1-uuid",           // 主 scope
    scopeName: "美租·装修托管",
    scopeKind: "project",         // 新增

    // 原有字段（主 scope 过滤）
    meetings, commitments, judgments, ...

    // 新增：关联维度摘要
    relatedThemes: [              // 该 project 下会议绑定的 topic scope
      { id, name: "资金结构", meetingCount: 12 },
      { id, name: "装修产品", meetingCount: 6 },
      ...
    ],
    relatedClients: [             // 该 project 下会议绑定的 client scope
      { id, name: "上海信托", meetingCount: 7 },
      { id, name: "宁波银行", meetingCount: 3 },
      ...
    ],
    crossScopeSignals: [          // 跨 scope 信号（来自知识轴）
      { kind: "concept_drift", fromScope: "T1-资金结构", signal: "..." },
      ...
    ]
  }
```

### 5.2 CEO 六房间 × 三维 scope 矩阵

| 房间           | 焦点 scope    | 关联维度       | 典型问题                                         |
| ------------ | ----------- | ---------- | -------------------------------------------- |
| 🧭 Compass   | **project** | topic 趋势   | "美租·装修托管"的战略方向是否偏离？资金结构主题的会议频次在增加——是焦虑还是机会？  |
| 🏛 Boardroom | **client**  | project 分布 | "万华"在装修托管 vs 租赁运营中的角色差异？"上海"区域的经营状况 vs "北京"？ |
| 🗼 Tower     | **project** | client 承诺  | "装修托管"对上海信托/宁波银行的承诺是否冲突？                     |
| ⚔️ War Room  | **topic**   | project 来源 | "供应链"问题在哪些 project 中出现？是共性问题还是个案？            |
| 📡 Situation | **client**  | topic 信号   | "宁波银行"关注的风控信号 vs "万华"关注的供应链信号——是否矛盾？         |
| 🪟 Balcony   | 任意          | 全部         | CEO 的时间在三个维度上的分配是否合理？                        |

### 5.3 Panorama 聚合增强

当前 `getPanoramaData` 接受 `scopeIds[]`。三维 scope 下：

```typescript
// 前端请求
GET /api/v1/ceo/panorama?primaryScope=P1-uuid&includeRelated=true

// 后端逻辑
1. 取 primaryScope 的六房间指标（现有逻辑）
2. JOIN mn_scope_members → 找到同会议的 client/topic scope
3. 为每个关联 scope 生成迷你 panorama（简化版指标）
4. 返回：
   {
     primary: { scopeId, name, kind, prisms: [...] },
     related: [
       { scopeId: "C5-uuid", name: "上海信托", kind: "client", miniPrisms: [...] },
       { scopeId: "T1-uuid", name: "资金结构", kind: "topic", miniPrisms: [...] },
     ],
     matrix: { ... }  // 交叉矩阵
   }
```

### 5.4 scopeMatcher 增强

当前 `scopeMatcher` 只匹配 scope name 的关键词/双字组。三维 scope 需要：

| 维度 | 匹配策略 | 示例 |
|------|---------|------|
| **project** | 关键词 + 业务实体识别 | "装修""托管""信托" → P1 |
| **topic** | 语义分类（零样本 or 规则） | 高频词含"保证金""SPV""担保" → T1 |
| **client** | 实体匹配（公司名/人名） | "万华""宁波银行" → C2/C4 |

## 6. 迁移方案

### 6.1 Phase 1：schema 调整（零停机）

| 步骤 | 操作 | SQL |
|------|------|-----|
| 1 | `mn_scope_members` 增加 `metadata` JSONB | `ALTER TABLE mn_scope_members ADD COLUMN metadata JSONB DEFAULT '{}';` |
| 2 | 现有 topic scope 的 `kind` 值不变，语义迁移 | 无 DDL |
| 3 | client scope 增加 `metadata.client_type` 区分内部/外部 | `UPDATE mn_scopes SET metadata = jsonb_set(metadata, '{client_type}', '"internal_region"') WHERE name IN ('上海','北京','成都');` |

### 6.2 Phase 2：scope 重建

| 操作 | SQL / API | 说明 |
|------|-----------|------|
| 创建 P1-P6 | `POST /scopes {kind: "project", ...}` | 6 个新 project scope |
| 创建 T1-T8 | `POST /scopes {kind: "topic", ...}` | 8 个新 topic scope（替换旧 5 个） |
| 创建 C1-C8 | `POST /scopes {kind: "client", ...}` | 8 个新 client scope（含 C1a/C1b/C1c 子 scope） |
| 重命名贝壳找房→贝壳·链家 | `PUT /scopes/:id {name, slug}` | slug: beike-lianjia |
| 上海/北京/成都调整 parent | `PUT /scopes/:id {parentScopeId: C1-uuid, slug: "beike-shanghai"}` | 保留为 C1 子 scope，更新 slug 和 metadata.client_type |
| 归档旧 scope | `PUT /scopes/:id {status: "archived"}` | 不删除，保留历史绑定 |
| 迁移绑定 | 批量 `POST /scopes/:id/bindings` | 按第 4 节映射表逐场绑定 |

### 6.3 Phase 3：代码适配

| 模块 | 改动 | 文件 |
|------|------|------|
| `loadPromptCtx` | 扩展返回 `relatedThemes` / `relatedClients` | `ceo/prompts/index.ts` |
| `scopeMatcher` | 增加语义分类逻辑 | `meeting-notes/runs/scopeMatcher.ts` |
| Panorama API | 支持 `primaryScope` + `includeRelated` | `ceo/panorama/service.ts` |
| 前端 scope 选择器 | 三维 Tab 切换 UI | `webapp/src/prototype/ceo/` |
| `PromptCtx` 类型 | 增加 `scopeKind`, `relatedThemes`, `relatedClients` | `ceo/prompts/types.ts` |

### 6.4 回滚

- 旧 scope 未删除（status=archived），可随时恢复
- 绑定是累加的，旧绑定和新绑定共存不冲突
- `mn_scope_members` 保留 `reason` 字段追踪绑定来源
