# 内容库 (content-library) workspace 权限隔离方案

> **状态**：已起草，待裁决（A / B / C）。
> **作者**：Claude Code 协作产出
> **依赖**：mn 模块 ws 隔离（迁移 032-039）+ ceo 模块 ws 隔离（迁移 041-043）已落地为成熟范式。
> **文档目的**：留方案、留决策上下文。当前**不动手**，等多租户上线时机或客户明确要求再启动。

---

## 1. 为什么需要这份方案

ceo / mn 已经走完一整套 ws 隔离（21 张 ceo_*、40+ 张 mn_*），content-library 整套 9 张 `content_*` + `task_purpose_docs` 全部没动 — 当时迁移 032 没把它们纳入。当前结果：

- A workspace 用户 GET `/api/v1/content-library/...` 任意接口可以看到 B workspace 用户喂进系统的事实 / 信念 / 实体关系。
- `ContentLibraryEngine.getOverviewStats()` 是全库聚合，跨 ws 数据混在一锅。
- 没有任何 router 挂 `preHandler` 守卫。

但 content-library 不像 ceo —— 它**架构上有意做全局知识图**：`content_entities.canonical_name` 全库 UNIQUE，所有租户的"巴菲特"应该解析到同一个 entity_id；`content_topic_enrichments` 是 LLM 调用过的议题缓存，跨租户复用反而是省钱省时间的。直接照抄 ceo 4-commit 范式会破坏这个语义。

**核心矛盾**：哪些表是"每个租户私有的内容"，哪些是"全平台共享的知识"，需要明确分桶 —— 这是一个**架构决策**，不是机械加列。

## 3. 三个选项

### A. 谨慎方案 — 只把"明显 per-tenant"的表 ws 隔离

**ws-bound 表**（4 张）：
- `content_facts`（已通过 asset/meeting 间接绑定，加显式 ws_id 收口）
- `content_production_log`（task-bound）
- `task_purpose_docs`（task-bound）
- `content_performance_history`（per-content 预测历史）

**保持无 ws_id（隐式 shared）**（6 张）：
- `content_entities`（全局规范化注册表，跨租户复用）
- `content_entity_relations`（实体图，跟随 entities 共享）
- `content_beliefs`（信念追踪，理论上全局；详见"开放问题"）
- `content_topic_enrichments`（LLM 增强缓存，省钱）
- `content_keyword_relations`（RSS 公共抓取的关键词聚合）
- `content_synthesis_cache`（综合缓存）

**优点**：
- 改动最小（4 张表 vs 10 张），耦合查询不变。
- 保留全局规范化语义。
- 跨租户复用 LLM 缓存，省钱。

**缺点 / 残留风险**：
- 聚合统计（`SELECT subject, COUNT(*) FROM content_facts` 之类）会让租户"数到"对方的存在性 — 行级数据被 ws filter 兜住，但 cardinality 暴露。
- `content_beliefs.supporting_facts UUID[]` 数组指向 facts，beliefs 是全局表 → 一条全局信念列出的"支撑事实"可能来自任意租户，列出来的人能看到 fact ID。是否泄露看 belief 的展示路径有没有再 JOIN facts；建议在 router 层做"二次过滤"（拿到 belief 后再按 facts.workspace_id 过滤 supporting_facts 数组）。

**实施量估算**：4 commit，~2 天工时。

### B. 激进方案 — 全部 10 张表 ws 隔离

**全部加 ws_id**，但通过 `is_shared=true` workspace 复用全局知识：

- 所有"全局规范化"表（entities / topic_enrichments / keyword_relations / synthesis_cache）的全局数据落到 slug='default' 且 `is_shared=true` 的 workspace。
- 每个租户写 entity / enrichment 时**先看 default ws 有没有，有就复用 ID（不重写）**；只有租户自己产生的、不该泄露的实体才写到自己 ws。
- 039 RLS policy 自动覆盖（policy 已经放行 `is_shared` ws）。

**优点**：
- 一致体验：每条 row 都有 ws_id，039 全覆盖，无认知例外。
- 聚合统计也能正确按 ws 过滤。

**缺点 / 风险**：
- `content_entities.canonical_name UNIQUE` 必须改成 `(canonical_name, workspace_id) UNIQUE`，或保持全局唯一靠"is_shared 单条 + per-ws 单条"约定 —— 后者代码层每次 entity 创建都要 `INSERT ... ON CONFLICT(canonical_name) WHERE workspace_id IN (default, current)` 这样的复合约束，**容易出 bug**。
- `entityResolver.ts:90` 的去重逻辑要重写（先看 default ws）。
- 16 处跨表 JOIN 全部要审查是否要带 ws filter（per-ws facts JOIN shared entities 其实没问题，但需要逐句确认）。
- 跨模块写（mn `persistClaudeFacts`）也要带上 ws_id —— 这点和 ceo runHandler 处理 mn_runs.workspace_id 是一样的范式，可借鉴。

**实施量估算**：8-10 commit，~1 周工时（含 entity_resolver 重写 + 集成测试）。

### C. 推迟（推荐）

**短期不动手**。理由：

1. **当前实质单租户**：content-library 整套抽取/规范化今天主要服务一个客户。多租户上线还没排上日程。
2. **过早做容易选错**：现在选 A 还是 B，取决于"未来多租户怎么用全局知识"——客户 X 想跨租户共享研究知识、客户 Y 不想，业务模型未定。
3. **已有兜底**：assets / tasks / mn_runs 的 ws_id 已经存在 → 应用层 SELECT 时如果加上 `JOIN assets WHERE workspace_id=$ws` 这种"间接过滤"，能兜住 90% 的泄露面。这条债**可以记账先记着**，不是立即起火。
4. **ceo / brain 已封口**：第二轮审计的 P0（brain router）已落地，ceo 端 ws 隔离闭环。content-library 是第二大块债，但**不在燃眉之急**。

**保留**：本文档作为决策入口，多租户上线 / 客户提诉求时再启动。届时按"4 触发条件"评估走 A 还是 B。

---

## 4. 推荐结论

**当前推荐：C（推迟）。**

如果接下来 1-3 个月内出现以下任一情况，立即启动方案 **A**（不直接跳 B，因为 A 是 B 的子集，A 做完再升 B 成本不变）：

- 第二个客户接入并使用 content-library 抽取功能。
- 安全/合规审计要求"内容数据租户级隔离"硬性指标。
- 客户在产品里看到了对方的 fact / 信念 / 议题热度（已发生泄露的事故）。

如果出现以下情况，再考虑升级到 **B**：

- 多租户实测下 A 方案的"cardinality 泄露"（聚合统计反推他人存在）成为客户投诉点。
- 业务方明确要"全租户隔离的知识图"（不复用全局 entity）。

## 6. 验证 / DoD（启动 A 时套用）

1. **迁移幂等**：044/045/046 跑两次不报错。
2. **Backfill 完整性**：4 张表 `SELECT count(*) WHERE workspace_id IS NULL = 0`。
3. **Trigger 兜底**：BEFORE INSERT 测试 — 不带 ws 仅带 asset_id 的 INSERT，落库 ws 自动从 assets 抄。
4. **跨 ws 行级守卫**：A ws 用户拿 B ws 的 factId 访问 GET/DELETE → 404，不暴露存在性。
5. **list 过滤**：A ws 调 `/content-library/facts?subject=X` 不返回 B ws 的 facts。
6. **跨模块写一致**：mn `persistClaudeFacts` 写入的 fact 自动落到对应 mn_runs.workspace_id。
7. **回归**：`npm run test` 全绿；`npm run build` / `tsc --noEmit` 干净。

## 7. 风险 / 回滚

| 风险 | 缓解 |
|---|---|
| backfill 把不该归 default 的旧数据全打到 default ws | A/B 选项均通过 037 trigger 兜底 + 043 同款"二次 backfill"防御；旧数据进 default 是历史事实，可接受 |
| RLS policy 误伤 SUPERUSER / api-key admin 路径 | 039 已 verified — 应用层 SUPERUSER bypass，preHandler 对 wsId=null 跳过 |
| 跨表 JOIN 漏改导致泄露 | A 方案改动小（4 张表，~25 处 SELECT），代码 review 可控；B 方案需逐处审 16 处 JOIN，建议 ultrareview |
| `entityResolver` 重写引入重复 entity（B 方案专属） | B-2 阶段必须配 lookup 测试 + ON CONFLICT 兜底 |

**回滚**：每个迁移配对应 `DROP COLUMN workspace_id` 或 `DISABLE ROW LEVEL SECURITY` 反向 SQL，commit message 留一行回滚命令。

## 9. 开放问题（启动前 owner 必答）

1. **`content_beliefs` 怎么走？** A 方案默认保持全局。如果产品要"租户级私有信念"，beliefs 也要进 A 列表（5 张表），但 `supporting_facts UUID[]` 数组的 ws 一致性需要应用层兜（trigger 救不了数组）。
2. **`content_topic_enrichments` 是 A 跳过还是 B 阶段才动？** 推荐 A 跳过 —— 它本质是 LLM 缓存，跨租户复用反而是收益。
3. **跨租户实体名字撞车怎么办？** B 方案下，租户 A 的 "巴菲特" 和租户 B 的 "巴菲特" 默认都映射到 default ws 的同一条 — 这是设计意图（规范化）。如果客户 Y 不接受，B-2 的 lookup 逻辑加一个 per-ws override flag。
4. **`content_synthesis_cache` schema 在哪建？** 仓库里搜不到 CREATE TABLE，可能 DB 手工建。启动前 ops 需要确认 schema 真相 + 加进迁移。
