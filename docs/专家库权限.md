# 专家库 (expert-library) workspace 权限隔离方案

> **状态**：已起草，**推荐立即启动**（与"内容库权限"不同 — 专家库 shared/per-ws 边界清晰，无架构争议）。
> **作者**：Claude Code 协作产出
> **依赖**：mn 模块 ws 隔离（迁移 032-039）+ ceo 模块 ws 隔离（迁移 041-043）已落地为成熟范式。
> **目的**：把第二大 ws 隔离债收口。先文档定型，等用户给开工信号。

---

## 1. 为什么需要做

ceo / mn 已经走完一整套 ws 隔离，专家库 7 张表里只有 3 张被 032 顺手覆盖（profiles / reviews / review_tasks），**其它 4 张完全裸奔**。当前结果：

- A workspace 调专家 `POST /api/v1/expert-library/invoke/:expertId` 留下的调用记录、用户评分、反馈，B workspace 的用户能直接看到。
- `expert_knowledge_sources`（专家知识源原文 — 包括客户访谈纪要）跨 ws 可见 — **这是直接的客户数据泄露面**。
- `expert_task_assignments`（哪个客户的哪个 task 调度给哪个专家、什么时候、deadline 多久）— 商业敏感的进度信息跨 ws 可见。
- 46 个 endpoint 里只有 4 个挂了 `authenticate`，没有任何 ws 守卫。

但和 ceo 不同 — 专家库**有清晰的"shared 知识"层**：codebase 里硬编码的 15 个公共专家 profile（巴菲特、芒格、乔布斯、马斯克等）是全平台共享的认知资源，每个租户都用同一个"巴菲特"做 LLM 角色。这部分应该明确归 `is_shared` workspace，不能跟着租户分裂。

**核心**：把 7 张表拆成"全局共享"和"租户私有"两层，全局层走 035 的 `is_shared` workspace 范式，私有层走 ceo 同款的 4-commit 模板。

> **隐患**：`expert_profiles` 在 `db/migrations/003`（schema A：user_id PK）和 `expert-library/001`（schema B：expert_id PK）两处都 `CREATE TABLE IF NOT EXISTS`。先跑的赢。当前线上数据库实际是哪个 schema 需要 ops 确认 — 本方案默认是 schema B（expert_id PK），因为 expert-library 全量代码都按 expert_id 写。**启动前必须先解决这个 schema 二义性**，否则 ws_id 加在哪个 schema 上是糊的。

### 2.2 表分类（明确 — 无争议）

| 类别 | 表 | 理由 |
|---|---|---|
| **全局共享**（落 `is_shared=true` 的 default ws） | `expert_profiles`<br>`expert_knowledge_sources` 中**built-in 专家的部分** | builtinExpertIds.ts 列出的 15 个专家是 codebase 资产；每个租户复用同一个"巴菲特"，不应该按租户分裂 |
| **租户私有**（per-ws，4-commit 范式） | `expert_invocations`<br>`expert_feedback`<br>`expert_task_assignments`<br>`expert_knowledge_sources` 中**用户上传部分** | 每条调用 / 反馈 / 调度都绑定具体 task / user，必须按 ws 隔离 |
| **已 ws-bound（032 已覆盖）** | `expert_reviews`、`expert_review_tasks` | task_id → tasks（已 ws-bound），037 trigger 兜底；本方案**不再涉及** |

`expert_knowledge_sources` 是混合表：built-in 专家的知识源（公开访谈、出版物）应该共享，租户私上传的知识源（自己采访的访谈、内部纪要）必须租户私有。**实施时按 expert_id 是否在 builtinExpertIds 里二分**。

## 3. 推荐方案 — 直接走 ceo 同款 4-commit

不像 content-library 需要 A/B 选择，专家库分类清晰，直接套范式。

### 3.1 四个 commit

| Commit | 改动 | 验证 |
|---|---|---|
| **1** | 新增 `migrations/044-expert-library-ws.sql`：给 4 张待治理表（`expert_knowledge_sources` / `expert_invocations` / `expert_feedback` / `expert_task_assignments`）加 nullable `workspace_id UUID REFERENCES workspaces(id)` + 索引；backfill：built-in 专家 / system-generated 行落到 default `is_shared=true` ws，其余落到 default ws | `SELECT count(*) WHERE workspace_id IS NULL` 全 0 |
| **2** | 新增 `migrations/045-expert-library-triggers.sql`：写 `inherit_ws_from_expert_invocation`（feedback.invoke_id → invocations）、复用 037 的 `inherit_ws_from_task`（task_assignments.task_id → tasks）。`knowledge_sources` 通过 `expert_id` 间接绑定（builtin 专家 → is_shared，自定义专家 → 037 通用 trigger 走不通，需显式传）。<br>同时改写入站点：`ExpertEngine.invoke*`、`feedbackLoop`、`knowledgeService`、`schedulingService` 的 INSERT 显式带 `currentWorkspaceId(request)` 或父表抄 | trigger 测试：feedback INSERT 不带 ws → 自动从 invocation 抄 |
| **3** | 新增 `modules/expert-library/workspaceGuard.ts`（仿 ceo `workspaceGuard.ts`）：URL `:expertId` / `:invocationId` / `:knowledgeId` / `:assignmentId` / `:feedbackId` 行级守卫；`router.ts` 顶部 `addHook('preHandler', expertWorkspaceGuard)`。<br>4 张表的 SELECT/list 加 `WHERE (workspace_id = $X OR workspace_id IN (SELECT id FROM workspaces WHERE is_shared))` 过滤（约 15-20 处 SELECT，按文件 batch）| A ws 用户访问 B ws 的 invocation 返回 404；built-in 专家任意 ws 仍可读 |
| **4** | 新增 `migrations/046-expert-library-not-null.sql`：4 张表 `SET NOT NULL` + 加 RLS policy（仿 043 通用扫描）；同时**先解决 expert_profiles schema 二义性**（确认 schema 真相 + 把 003 老 schema 标记 deprecated 或 drop） | NULL 二次校验、RLS policy 落盘、profiles schema 单一定义 |

### 3.2 边界处理 — built-in 专家路径

启动 commit 1 之前，先决定 builtin 专家归属：

```typescript
// expertSeed.ts / ExpertEngine.upsertProfile：判定 is_shared 写入
import { COLLECT_BUILTIN_EXPERT_IDS } from './builtinExpertIds.js';
const SHARED_IDS = COLLECT_BUILTIN_EXPERT_IDS();
const isBuiltin = SHARED_IDS.includes(expertId);
const targetWsId = isBuiltin ? defaultIsSharedWsId : currentWorkspaceId(request);
```

`knowledgeService.addSource(expertId, ...)` 同款判定 — built-in 专家的知识源默认归 is_shared，租户自定义专家的知识源归当前 ws。

## 4. 验证 / DoD

1. **迁移幂等**：044/045/046 跑两次不报错。
2. **Backfill 完整性**：4 张表 `SELECT count(*) WHERE workspace_id IS NULL = 0`；built-in 专家相关行（用 `expert_id IN (SELECT expert_id FROM expert_profiles WHERE workspace_id IN (SELECT id FROM workspaces WHERE is_shared))`）全部归 is_shared。
3. **Trigger 兜底**：BEFORE INSERT 测试 — feedback / task_assignments 不带 ws，从父表自动抄。
4. **跨 ws 行级守卫**：A ws 用户拿 B ws 的 invocationId 访问 GET/PATCH → 404。
5. **list 过滤**：`/expert-library/invocations/recent?expertId=巴菲特` 在两个 ws 看到的列表互不交叉，但**built-in 专家本身**在两个 ws 都能查到。
6. **跨模块读一致**：`mn/expertProfileLoader` 读 builtin 专家正常；`content-library/crossSourceSearch` 读 invocations 时按调用方 ws 过滤。
7. **回归**：`npm run test` 全绿；`tsc --noEmit` 干净。
8. **Schema 单一性（前置）**：`expert_profiles` 仅一处 schema 定义生效（drop 或 noop 老的 003 定义）；ops 确认线上 DB 列结构与 `001-expert-library.sql` 对齐。

## 5. 风险 / 回滚

| 风险 | 缓解 |
|---|---|
| `expert_profiles` schema 二义性，ws_id 加在错误 schema | commit 4 前置：ops 跑 `\d expert_profiles` 确认列结构；如有冲突先开 schema 收敛 PR |
| built-in 专家 backfill 漏判，被打到具体 ws → 跨 ws 用户看不到 | backfill SQL 用 `WHERE expert_id = ANY($1::text[])` 显式带 builtinExpertIds 数组，code review + 集成测试兜底 |
| user_id 字段在某些表中（expert_profiles 003 schema、invocations）— 是否需要 user-level 隔离 | 本方案**不引入 user-level 过滤**；只做 ws-level；user 维度由产品后续决策 |
| 跨模块写漏改 | 改动 commit 2 检查清单：8 处 INSERT（参见 2.3），逐处确认 |
| 集成测试缺失 | 本方案**自带集成测试 commit**（可选合到 commit 3 或独立 commit 5）|

**回滚**：每个迁移配对应 `DROP COLUMN workspace_id` 或 `DISABLE ROW LEVEL SECURITY` 反向 SQL，commit message 留一行回滚命令。

## 6. 不在本方案范围

- ❌ user-level 隔离（同 ws 内 user A 看不到 user B 的调用记录）— 用例不明，留作后续。
- ❌ `expert_profiles` schema 二义性彻底治理（drop 003 老 schema）— 如不构成阻塞，并行处理；构成阻塞则前置一个 schema 收敛 commit。
- ❌ 老旧 `experts`、`expert_library`（`db/connection.ts:653`）等"跟随 v3.x 残留"表 — 看是否还有读写依赖再决定 drop / migrate。
- ❌ 前端改动 — webapp 调专家库自动带 cookie ws，无需额外 header。
- ❌ 内容库（content-library）—— 单独成案 `docs/内容库权限.md`。

## 7. 开放问题（启动前 owner 必答）

1. **`expert_profiles` schema 真相**：线上 DB 是 003 schema 还是 001-expert-library schema？需 ops 跑一次 `\d expert_profiles` 给出确认。
2. **built-in 专家边界**：`builtinExpertIds.ts` 列了 15 个，是否还有其他 codebase 引入但漏列的？启动 commit 1 前 grep 一遍 `expert_id =` 确保覆盖完整。
3. **`debateEngine.ts:225` runtime 创造的临时专家**：归 per-ws 还是 is_shared？建议 per-ws — 谁创造谁拥有，避免临时实验污染共享池。
4. **`expert_knowledge_sources` 内的 built-in 专家访谈**：是否真的应该 is_shared？某些访谈可能是采购第三方版权资源，理论上跨租户复用是合规的；如果是某客户独家访谈则必须 per-ws。**启动前盘一次现存数据归属**。
5. **`writer.ts:395` agent 写 expert_profiles**：什么场景会写？是否触发 schema 二义性问题？

## 9. 与"内容库权限.md"的关系

| 维度 | 内容库 | 专家库（本方案） |
|---|---|---|
| Per-ws / shared 边界 | **模糊** — content_entities / topic_enrichments 全局规范化是设计意图 | **清晰** — built-in 专家 vs 用户专家两层泾渭分明 |
| 表数量 | 9 张 + 1 张 schema 真相未明 | 4 张待治理（profiles 已 ws，外加 schema 二义性兜底） |
| 推荐 | C（推迟，等多租户上线再决策 A/B）| **直接做** A 同款（无架构争议） |
| 工时估算 | A: 2 天 / B: 1 周 | 4 commit / ~2 天 |
| 商业敏感泄露面 | 中 — content_facts / production_log 是抽取出来的二级数据 | **高** — knowledge_sources 是客户原始访谈纪要、task_assignments 是商业进度 |

**优先级**：本方案应优先于内容库执行 — 直接对客户原始数据的隔离债。
